# M13 — Git LFS: Batch API + transfer + verify

**Status:** approved (2026-05-18)
**Roadmap row:** `docs/superpowers/specs/2026-05-03-bucketvcs-oss-decomposition-design.md:93`
**Original-spec reference:** §22 in `docs/original-spec.md:1185`

## 1. Scope

Implement Git LFS so a stock `git-lfs` client can push and pull large objects against a bucketvcs gateway over both HTTPS and SSH.

In scope:

- LFS Batch API (`POST .../info/lfs/objects/batch`) for `upload` and `download` operations.
- `basic` transfer adapter only.
- Direct transfer via signed object-store URLs (S3 / R2 / GCS / Azure); proxied HMAC URL fallback for the localfs backend.
- Verify action (`POST .../info/lfs/objects/<oid>/verify`).
- SSH `git-lfs-authenticate` command that issues a short-TTL bearer for HTTPS transfer.
- Per-repo storage layout under the existing repo prefix.
- Auth and authorization via the existing M4 token + repo-permission model.
- Operator guide and MinIO smoke test on the M11 pattern.

Out of scope for M13 (explicit non-goals):

- LFS Locking API.
- Multipart upload / custom transfer adapters.
- Reachability-based LFS GC (LFS objects are removed only when the repo is removed).
- LFS byte quotas, per-tenant LFS pooling, or cross-repo dedupe.
- LFS-specific token scopes.

These are listed because their absence affects operator expectations; each is recorded as deferred work in §11.

## 2. Architecture

```
                ┌────────────────────────────────────────┐
                │  git-lfs client                        │
                └─────────┬───────────┬──────────────────┘
                          │HTTPS      │SSH (auth only)
                          ▼           ▼
        ┌─────────────────────────────────────────┐
        │  gateway.Server (existing mux)          │
        │   /<owner>/<repo>.git/info/lfs/         │
        │     objects/batch         (POST)        │
        │     objects/<oid>/verify  (POST)        │
        │   /<owner>/<repo>.git/lfs/objects/<oid> │
        │     (localfs proxied PUT/GET)           │
        │   /_bundle/, /_pack/, /healthz, …       │
        └──────────────┬──────────────────────────┘
                       │
                       ▼
        ┌─────────────────────────────────────────┐
        │  internal/lfs/                          │
        │   batch.go     parse + build response   │
        │   transfer.go  presign / proxied URL    │
        │   verify.go    HEAD + size check + log  │
        │   store.go     Head / Presign{Put,Get}  │
        │   auth.go      git-lfs-authenticate     │
        │   handler.go   HTTP wiring              │
        └──────────────┬──────────────────────────┘
                       │
              ┌────────┴────────┐
              ▼                 ▼
       storage.ObjectStore   internal/proxiedurl
       (S3/R2/GCS/Azure)     (HMAC URL for localfs)
              │
              ▼
     <repo-prefix>/lfs/objects/<sha256>
```

`internal/sshd/` gains one new command handler. When a client invokes `git-lfs-authenticate <owner>/<repo> <upload|download>`, the SSH session authenticates by SSH key (existing M4 flow), authorizes against the named repo, and issues a short-TTL bearer bound to user + repo + op. The reply is the LFS standard JSON `{href, header{Authorization}, expires_at}` written to stdout. The client then uses that bearer for the HTTPS Batch API and signed-URL transfers.

## 3. Components and interfaces

```go
// internal/lfs/store.go — scoped to a single repo's LFS area.
type Store struct {
    backend storage.ObjectStore
    prefix  string // "<repo-prefix>/lfs/objects/"
}

func (s *Store) Key(oid string) string
func (s *Store) Head(ctx context.Context, oid string) (size int64, exists bool, err error)
func (s *Store) PresignPut(ctx context.Context, oid string, size int64, ttl time.Duration) (url string, headers http.Header, err error)
func (s *Store) PresignGet(ctx context.Context, oid string, ttl time.Duration) (url string, headers http.Header, err error)

// Only when the backend has no native presign (localfs):
func (s *Store) ProxiedPutURL(oid string, size int64, ttl time.Duration) (url string, headers http.Header)
func (s *Store) ProxiedGetURL(oid string, ttl time.Duration) (url string, headers http.Header)

// internal/lfs/batch.go — pure protocol logic, no I/O.
type ObjectRef    struct{ OID string; Size int64 }
type ObjectAction struct {
    OID, Error string
    Size       int64
    Actions    map[string]Action // "upload" | "download" | "verify"
}
type Action struct {
    Href     string
    Header   map[string]string
    ExpiresAt time.Time
}
type Request  struct{ Operation string; Objects []ObjectRef; Transfers []string }
type Response struct{ Transfer string; Objects []ObjectAction }

func Build(ctx context.Context, req Request, store *Store, verifyBaseURL string, bearerForVerify string) (Response, error)

// internal/lfs/verify.go
func Verify(ctx context.Context, store *Store, oid string, claimedSize int64) error

// internal/lfs/auth.go — SSH-side bearer issuance.
type SSHAuthResponse struct {
    Href      string            `json:"href"`
    Header    map[string]string `json:"header"`
    ExpiresAt time.Time         `json:"expires_at"`
}
func IssueSSHToken(ctx context.Context, issuer TokenIssuer, user, repo, op string, ttl time.Duration) (SSHAuthResponse, error)

// internal/lfs/handler.go — wires routes; constructed by gateway.Server.
type Deps struct {
    AuthMiddleware func(http.Handler) http.Handler
    NewStore       func(repoPrefix string) *Store
    ResolveRepo    func(*http.Request) (owner, repo, prefix string, perms RepoPerms, err error)
    PresignTTL     time.Duration
    Bearer         func(*http.Request) string // forwarded into verify-action Authorization
    Logger         *slog.Logger
}
func NewHTTPHandler(d Deps) http.Handler
```

Reused systems and the reason for each:

| Reused | Where | Why |
|---|---|---|
| Auth (`internal/auth`) | M4 token + repo perms via existing middleware | Single token model across Git, bundle/pack URI, and LFS. No new scope. |
| Storage presign | M11 `ObjectStore.PresignPut/Get` | LFS direct mode is the same presign primitive as M11 packfile-uri. |
| Proxied HMAC URL | M11 `internal/proxiedurl` | Localfs has no native presign; the same HMAC scheme used for proxied packs/bundles also serves LFS uploads and downloads when the backend is localfs. |
| sshd command dispatch | M6 `internal/sshd` | One additional command branch (`git-lfs-authenticate`) alongside the existing Git command handling. |

## 4. Storage layout

Each repo's LFS area lives at:

```
<repo-prefix>/lfs/objects/<sha256>
```

`<repo-prefix>` is the existing per-repo prefix used by Git pack data. Objects are content-addressed by their SHA-256 OID (the LFS pointer field). No size suffix, no per-byte sharding (LFS clients do not need it; deep prefixes complicate listing without buying anything at OSS scale).

Single PUT per object. Operator-facing single-object size limits track the backend's single-PUT limit:

| Backend | Single PUT limit |
|---|---|
| S3 / R2 | 5 GB |
| GCS | 5 TB |
| Azure Blob | ~190 GB (block blob) |
| localfs | bounded by gateway memory headroom; documented as "use S3/R2/GCS/Azure for any LFS workload over a few hundred MB" |

These limits are documented in the operator guide and surfaced as an error response when a Batch request claims a size that exceeds the backend's documented single-PUT limit.

## 5. Wire formats and flows

### 5.1 Upload

```
client:  POST .../info/lfs/objects/batch
         Authorization: <token>
         { "operation":"upload", "transfers":["basic"],
           "objects":[{"oid":"...","size":N}, ...] }

gateway: authn (token) → authz (write perm on repo) → for each oid:
           Head(oid)
             exists, size matches    → { oid, size, actions:{} }
             exists, size mismatch   → { oid, size, error:{code:422,message:"size mismatch"} }
             missing                 → { oid, size,
                                         actions:{
                                           upload: { href:<presigned PUT>, header:{...},
                                                     expires_at:<RFC3339> },
                                           verify: { href:".../info/lfs/objects/<oid>/verify",
                                                     header:{Authorization:<bearer>} } } }

client:  PUT <upload.href> (bytes) — bucket OR proxied gateway URL on localfs.
client:  POST <verify.href>  { "oid":"...", "size":N }
gateway: Head(oid); size matches → 200; else 422 / 404. Emit audit event.
```

### 5.2 Download

```
client:  POST .../info/lfs/objects/batch
         { "operation":"download", "transfers":["basic"],
           "objects":[{"oid":"...","size":N}, ...] }

gateway: authn → authz (read) → for each oid:
           Head(oid)
             exists → { oid, size, actions:{ download:{ href:<presigned GET>, header,
                                                        expires_at } } }
             missing → { oid, size, error:{ code:404, message:"object not found" } }

client:  GET <download.href> — direct from bucket OR proxied URL on localfs.
```

### 5.3 SSH authenticate

```
client:  ssh git@host git-lfs-authenticate <owner>/<repo> <upload|download>
sshd:    authn (SSH key) → authz (perm matches op) → issue short-TTL bearer.
sshd:    stdout:
           { "href":  "https://host/<owner>/<repo>.git/info/lfs",
             "header":{ "Authorization":"Bearer <jwt>" },
             "expires_at":"<RFC3339>" }
```

The client then drives Batch + transfer over HTTPS using that bearer.

## 6. Error handling

Per-request top-level errors (HTTP status + LFS error JSON `{"message":"...","documentation_url":"..."}`):

| Condition | Status |
|---|---|
| Missing / invalid token | 401, `WWW-Authenticate: Basic` |
| Token valid, lacks perm | 403 |
| Repo not found | 404 |
| Malformed JSON / missing required field | 422 |
| `transfers` does not include `"basic"` | 422 |
| Internal / presign failure | 500 |

Per-object errors (top-level 200, `error` set on each affected object) for: object missing on download, size mismatch on upload-existing, backend HEAD failure on a single object.

Verify errors:

| Condition | Status |
|---|---|
| Object missing in storage | 404 |
| Stored size ≠ claimed size | 422 |
| Backend error | 500 |

## 7. Observability

Metrics (additive to the M9/M11 metric set):

```
lfs_batch_requests_total{op="upload|download",result="ok|unauthorized|forbidden|notfound|error"}
lfs_batch_objects_total{op,result="new|exists|missing|error"}
lfs_verify_requests_total{result="ok|missing|size_mismatch|error"}
lfs_ssh_authenticate_total{op,result="ok|denied|error"}
lfs_presign_seconds{backend="s3|r2|gcs|azure|localfs_proxied"} (histogram)
```

Audit events (slog, flat-attrs, same shape as M11):

```
event=lfs.batch             repo=… user=… op=upload|download n_objects=… result=…
event=lfs.verify            repo=… user=… oid=… size=…              result=…
event=lfs.ssh_authenticate  repo=… user=… op=… ttl_seconds=…        result=…
```

No bytes-transferred metric. In direct mode the gateway never sees the bytes; in proxied mode the count would be local-only signal. The operator guide documents the path for getting byte usage from the object store directly (S3 access logs, GCS audit logs, etc.).

## 8. CLI surface

No new top-level subcommands. LFS is part of `bucketvcs serve`:

```
bucketvcs serve
  --lfs                                   # default: true
  --lfs-presign-ttl=15m                   # default
  --lfs-ssh-token-ttl=15m                 # default
```

LFS routes are mounted unconditionally when `--lfs=true`. There is no per-repo LFS toggle in M13 — LFS objects appear in storage only when a client pushes them, so there is no storage cost for repos that do not use LFS.

## 9. Testing

| Layer | Coverage |
|---|---|
| Unit | `Build()` request → response shape, per-object branches (exists, exists+size-mismatch, missing, error), download missing-object path, transfer-adapter rejection, malformed-request handling. Fake `Store`. |
| Unit | `Verify()` for size match, size mismatch, missing, backend error. |
| Unit | `IssueSSHToken()` TTL and scope binding. |
| Integration | Full batch + presigned upload + verify + presigned download over httptest gateway against localfs (proxied URLs). |
| Storage conformance | Each backend's `PresignPut`/`PresignGet` exercised through `lfs.Store` in the existing storage conformance harness. |
| Smoke (localfs) | `scripts/m13-lfs-smoke-local.sh` — stock `git-lfs` push and pull. |
| Smoke (MinIO) | `scripts/m13-lfs-smoke-minio.sh` — stock `git-lfs` push and pull, asserting `X-Amz-Signature` present in client trace. |
| SSH | `git-lfs-authenticate` over a real SSH session; the issued bearer drives a working Batch + transfer round trip. |

Strong-marker testing (same posture as M11): positive markers (`X-Amz-Signature`, `Authorization: Bearer` in SSH-authenticate response, verify-success audit line) and negative markers (no `upload` action when the object already exists at the claimed size; no `download` action present on an upload-op response).

## 11. Deferred work

- LFS Locking API (M13.x or later milestone).
- Multipart upload via custom transfer adapter (M13.x if real workloads need it).
- Reachability-based LFS GC: scan manifests for pointer blobs, mark referenced OIDs, sweep unreachable LFS objects after retention. Today, LFS objects only disappear when the repo is removed (which sweeps the prefix). Listed as future work in the operator guide.
- LFS byte quotas, per-tenant pooling, cross-repo dedupe.
- LFS-specific token scopes (e.g., LFS-pull-only CI tokens).

## 12. Risks

- **Stock `git-lfs` interop edge cases.** Mitigated by exercising the real binary in smoke tests, not a curl-faked wire format.
- **Per-backend presign quirks** (Azure SAS shape, GCS V4 signing nuance). M11 already shook these out and the same `ObjectStore.PresignPut/Get` methods are reused unchanged.
- **Localfs proxied PUT bodies.** The same proxied-PUT machinery already used in M11 for proxied packs is reused here; body-size limits and error handling are inherited.
- **Verify HEAD before the object is durably visible.** On eventually-consistent backends, an immediate verify after PUT may HEAD-miss. S3 and GCS now offer strong read-after-write for new objects; for the localfs and Azure paths used in tests, post-PUT visibility is immediate. Documented in the operator guide.

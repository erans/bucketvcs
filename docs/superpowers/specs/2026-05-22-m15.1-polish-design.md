# M15.1: Polish — repo delete, actor override, secret rotation, storage_backend population

**Status:** Design.
**Date:** 2026-05-22.
**Scope:** Close 4 of M15's documented deferrals in a tight follow-on milestone.

## 1. Goals

Ship four small, mechanical improvements that close the loop on M15:

1. `bucketvcs repo delete` CLI + `EventRepoDeleted` emitter (completes 7 of 8 events in the M15 taxonomy)
2. `--actor=<string>` override on repo CLI subcommands (closes M15-noted privacy leak)
3. `bucketvcs webhook endpoint rotate-secret` CLI (compromised-secret recovery without losing endpoint history)
4. `ObjectStore.Name() string` interface method + receivepack wiring (fills the empty `PushPayload.StorageBackend` field)

### 1.1 Out of scope (deferred to a future milestone)

- **`repo rename`** — bucketvcs storage paths are derived from `(tenant, repo)` with no indirection layer. Real rename needs either a `storage_prefix` column in the `repos` table + gateway routing changes, or copy+delete across the storage prefix. Substantial; not polish.
- **`webhook prune`** — useful for table-size management but no immediate operator pain documented.
- **`delivery show --format=json`** — workaround exists (`delivery list --format=json` + filter).
- **`EventRepoRenamed` emitter** — depends on `repo rename` shipping.

## 2. Item 1: `bucketvcs repo delete`

### 2.1 CLI surface

```
bucketvcs repo delete --auth-db=<path> --tenant=<t> --repo=<r>
                      [--purge-storage] [--actor=<string>]
```

Exit codes: 0 ok, 1 operational error (db unreachable, `ErrNoSuchRepo`, partial purge), 2 usage error (missing flags).

### 2.2 Behavior

1. Open authdb via `sqlitestore.Open(*authDB)`.
2. Construct `webhooks.Service` from the same db handle for the event emission.
3. Call `store.DeleteRepo(ctx, *tenant, *repo)`. The sqlite FK cascade (CASCADE on `tenant, repo` references) handles dependent rows in:
   - `webhook_endpoints` (M15 migration 0006)
   - `protected_refs` (M14 migration 0005)
   - `quotas` (M13.5 migration 0004 — keyed on tenant only, so deletion of the last repo for a tenant does NOT remove the quota row; that's correct since quota is tenant-level)
   - `lfs_locks` (M13.3 migration 0003)
   - Any other tables with `(tenant, repo)` FK → cascade
4. If `--purge-storage`:
   - Open the storage backend from the config the same way `bucketvcs serve` does (read storage URI from env or flag — match existing CLI pattern in `cmd/bucketvcs/repocmd.go`)
   - Iterate `objectStore.List(ctx, "<tenant>/<repo>/", &ListOptions{Limit: 1000})` paginated via `ListPage.NextToken`
   - For each key returned, call `objectStore.DeleteIfVersionMatches(ctx, key, objectVersion)`. The version comes from the List response.
   - Track `purged` (deletes succeeded) and `errors` (deletes failed). Continue on per-key errors — log via slog but don't abort.
   - Emit a single summary line at the end: `purged=N keys errors=M (M>0 leaves orphans; re-run safe via idempotent DeleteIfVersionMatches)`
5. Emit `EventRepoDeleted` webhook (always, regardless of `--purge-storage` outcome) via `webhookSvc.Enqueue`. Payload is `RepoLifecyclePayload{}` (envelope alone). Fail-open per M15 pattern.
6. Emit audit event `repo.deleted` with attrs `{tenant, repo, actor, purge_storage_requested, purged_keys, purge_errors}`. The audit event is the canonical record for operators; webhook is best-effort notification.

### 2.3 Error semantics

| Failure | Behavior |
| --- | --- |
| `--auth-db`, `--tenant`, `--repo` missing | exit 2, usage message |
| `sqlitestore.Open` fails | exit 1, error to stderr |
| `Store.DeleteRepo` returns `ErrNoSuchRepo` | exit 1, "no such repo" message; no webhook emission |
| `Store.DeleteRepo` returns other error | exit 1, raw error to stderr |
| `--purge-storage` listing fails | exit 1 ONLY if the first List call fails (no progress made). If pagination succeeds partway then fails, log the error, continue with remaining pages, exit 1 with a count summary |
| `DeleteIfVersionMatches` returns version-conflict | log as warning, continue (someone wrote to this key concurrently with the purge; safe to leave) |
| Webhook enqueue fails | log via `webhooks.EmitEnqueueFailed`, do NOT propagate (consistent with M15 fail-open) |

### 2.4 Operator UX

```
$ bucketvcs repo delete --auth-db=auth.db --tenant=acme --repo=site
tenant=acme  repo=site  deleted

$ bucketvcs repo delete --auth-db=auth.db --tenant=acme --repo=site --purge-storage --actor=ops-cron
tenant=acme  repo=site  deleted  purged=247 keys  errors=0
```

## 3. Item 2: `--actor=<string>` flag

### 3.1 Surface

Add `--actor=<string>` to:
- `bucketvcs repo register`
- `bucketvcs repo grant`
- `bucketvcs repo revoke`
- `bucketvcs repo public`
- `bucketvcs repo delete` (new in this milestone)

Default: `cliActor()` (existing helper returning OS username via `os/user.Current()`). When `--actor` is specified, validate non-empty (reject empty string with exit 2), then use that value.

### 3.2 Threading

The actor value already flows through to `webhooks.EmitEnqueueFailed`, `webhookSvc.Enqueue` (envelope `actor` field), and existing audit emissions in `cmd/bucketvcs/repocmd.go`. The change is purely flag plumbing — read from flag-or-default at the top of each handler, pass through unchanged.

### 3.3 Backwards compatibility

Operators who don't pass `--actor` see identical behavior to today. Webhook receivers who were filtering or alerting on OS-username actor values continue to receive them by default; they only see different values when an operator explicitly opts in via `--actor`.

## 4. Item 3: `bucketvcs webhook endpoint rotate-secret`

### 4.1 CLI surface

```
bucketvcs webhook endpoint rotate-secret --auth-db=<path> --id=<N>
```

### 4.2 Service API

Add to `internal/webhooks/service.go`:

```go
// RotateSecret generates a new 32-byte random secret for the endpoint and
// returns it. Existing in_flight deliveries continue with the secret they
// captured at claim time; pending deliveries pick up the new secret on
// next claim. Returns ErrNotFound if no endpoint matches.
//
// Operators who need strict cutover (no overlap window where both old and
// new secrets validate) should Disable the endpoint first, wait for queue
// drain, then call RotateSecret and Enable.
func (s *Service) RotateSecret(ctx context.Context, id int64) (string, error)
```

Implementation:
1. Call existing `generateSecret()` for the new secret string
2. `UPDATE webhook_endpoints SET secret=? WHERE id=?` with RowsAffected check
3. If RowsAffected == 0, return `ErrNotFound`
4. Return new secret on success

### 4.3 CLI behavior

1. Parse `--auth-db` and `--id`. Both required.
2. Open Service via existing `openWebhookSvc(*authDB)`.
3. Call `svc.RotateSecret(ctx, *id)`.
4. On `ErrNotFound`: exit 2 (mirrors `delivery replay` ErrNotFound treatment from M15 round 3 fixes).
5. On other error: exit 1.
6. On success: print one line in the same format as `endpoint add`:
   ```
   endpoint_id=42  rotated
   secret=<new-43-char-base64url>   # store this now — it will not be shown again
   ```
7. Emit audit event `webhooks.endpoint_secret_rotated` with attrs `{endpoint_id, tenant, repo, actor}`. The actor comes from the standard `--actor` flag (mirroring repo subcommands — add `--actor` to this command too for consistency with item 2's pattern).

### 4.4 Delivery semantics

Immediate cutover. The `claim()` function in `worker.go` reads `e.secret` at SELECT time, so:
- Rows already in `in_flight` use whatever secret was captured at their claim
- Rows still `pending` will be signed with the new secret on the next worker tick

Operators are responsible for orchestrating strict cutover via Disable+drain+rotate+Enable if their receiver can't tolerate both old and new secrets being valid for a few seconds.

## 5. Item 4: `ObjectStore.Name() string`

### 5.1 Interface change

Add to `internal/storage/objectstore.go`:

```go
// Name returns the canonical backend kind: "localfs", "s3compat", "gcs",
// "azureblob". Used by receivepack to populate PushPayload.StorageBackend
// (spec §24) and by operator-facing logs.
//
// Implementations MUST return one of the documented kind strings; new
// backends should pick a stable lowercase identifier and document it in
// the storage backend support matrix (spec §11.1).
Name() string
```

### 5.2 Backend implementations

| Backend | File | Returns |
| --- | --- | --- |
| Local filesystem | `internal/storage/localfs/localfs.go` | `"localfs"` |
| S3-compatible (R2, S3, MinIO) | `internal/storage/s3compat/s3compat.go` | `"s3compat"` |
| Google Cloud Storage | `internal/storage/gcs/gcs.go` | `"gcs"` |
| Azure Blob | `internal/storage/azureblob/azureblob.go` | `"azureblob"` |

Each `Name()` is a one-line method on the existing Store struct:

```go
func (s *Store) Name() string { return "localfs" }
```

### 5.3 Test-mock implementations

Any in-tree test helper that implements `ObjectStore` will fail to compile until it adds `Name()`. The fix is mechanical (add a trivial method returning a test-specific kind string like `"memstore"` or `"test"`). Locate via:

```bash
go build ./...
# Will print compile errors for any type that doesn't satisfy ObjectStore.
```

### 5.4 Receivepack wiring

In `internal/gitproto/receivepack/complete.go` find the existing push-emission site (M15 Task 5 added it). The current code passes `storageBackendName` which is either empty or an undefined identifier. Replace with:

```go
storageBackend := ""
if eng.ObjectStore != nil {
    storageBackend = eng.ObjectStore.Name()
}
payload := webhooks.PushPayload{
    // ...
    StorageBackend: storageBackend,
    // ...
}
```

(`eng.ObjectStore` is the `ObjectStore` already on `EngineRequest` per pre-M15 wiring. If it's currently named something else, use the actual field name.)

### 5.5 Conformance suite update

`internal/storage/conformance/` tests every backend against a contract. Add one trivial assertion to the suite:

```go
func testName(t *testing.T, s storage.ObjectStore) {
    if s.Name() == "" {
        t.Errorf("Name() returned empty string")
    }
}
```

Register the test in the conformance harness so all backends + future backends get it for free.

## 7. Failure modes

| Failure | Behavior |
| --- | --- |
| `repo delete` on non-existent | exit 1, `repo: no such repo acme/site` |
| `repo delete --purge-storage` storage backend unavailable | exit 1 BEFORE auth delete (preserves authdb consistency) — check ObjectStore is configured before calling DeleteRepo |
| `repo delete --purge-storage` partial list/delete failure | log per-error, summary line shows non-zero `errors=`, exit 1 (operators rerun to mop up) |
| `rotate-secret` on non-existent id | exit 2 (usage error: bad id) |
| `--actor=""` | exit 2 with "actor must be non-empty if specified" |
| `ObjectStore.Name()` returns empty | conformance test fails CI; runtime tolerates empty (StorageBackend field just stays empty in webhooks, matching M15 today's behavior) |
| Webhook enqueue fails after successful delete | fail-open audit `webhooks.enqueue_failed`; delete is canonical (authdb row gone) |

## 9. Implementation order

Tasks 1-4 are mostly independent. Recommended order optimizes for review surface:

1. **Task 1: ObjectStore.Name() + conformance + receivepack wiring** — touches the most files (5 backends + interface + receivepack + any test mocks), but each touch is 1-3 lines. Get the compile-break-and-fix cycle done early.
2. **Task 2: `--actor=` flag on all 5 repo subcommands** — pure plumbing in `cmd/bucketvcs/repocmd.go`. Smallest surface.
3. **Task 3: `webhook endpoint rotate-secret`** — Service method + CLI subcommand + audit event. Self-contained inside the webhooks package + CLI.
4. **Task 4: `repo delete` CLI + EventRepoDeleted emitter + smoke extension** — largest surface (touches authdb, storage, webhooks, audit, smoke). Goes last so the prior items are stable when the smoke gets extended.

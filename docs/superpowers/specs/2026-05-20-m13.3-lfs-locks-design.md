# M13.3 — LFS Locks API design

**Status:** draft (under review)
**Date:** 2026-05-20
**Predecessor:** M13 (LFS HTTPS+SSH, commit 1e9ed58)
**Tag target:** `m13.3-lfs-locks`

## Goal

Implement the Git LFS Locking API ([spec](https://github.com/git-lfs/git-lfs/blob/main/docs/api/locking.md)). Closes operator-guide §8.1 deferred item. After this milestone:

- `git lfs lock <path>` creates a lock and returns its ID.
- `git lfs locks` lists locks, paginated, filterable by path/ID/cursor.
- `git lfs unlock <path>` deletes a lock; `--force` is honored only for write-permitted users on locks they do not own.
- `git lfs push` calls `/locks/verify` to learn which paths are locked by others and refuses to push them (client-side enforcement; the server returns the partition).
- Locks survive process restart (durable in sqlite).
- Locks are scoped per-repo and optionally per-ref.

Out of scope for M13.3: lock TTLs, push-blocking enforcement (LFS clients enforce client-side per the spec — the server's job is only to return the verify partition correctly), SSH-native lock transport.

## Architecture

### Storage

A new `lfs_locks` table in the existing M4 authdb sqlite file (`--auth-db`). No new operator flag.

```sql
CREATE TABLE IF NOT EXISTS lfs_locks (
    id            TEXT PRIMARY KEY,           -- server-assigned, ULID-style
    tenant        TEXT NOT NULL,
    repo          TEXT NOT NULL,
    path          TEXT NOT NULL,              -- absolute repo-relative path
    ref_name      TEXT,                       -- nullable; null = repo-wide
    owner_user_id TEXT NOT NULL,              -- M4 users.id
    locked_at     INTEGER NOT NULL,           -- unix seconds
    UNIQUE(tenant, repo, path)                -- one lock per path per repo
);
CREATE INDEX IF NOT EXISTS idx_lfs_locks_tenant_repo ON lfs_locks(tenant, repo);
CREATE INDEX IF NOT EXISTS idx_lfs_locks_owner ON lfs_locks(owner_user_id);
```

The `UNIQUE(tenant, repo, path)` constraint enforces the LFS-spec "lock already exists" semantics at the DB layer. Conflicting inserts surface as `sqlite3.ErrConstraintUnique`, which the handler maps to HTTP 409 with the existing lock's metadata (per spec).

The schema is created lazily by `sqlitestore.Open` (matching how the users / tokens tables are bootstrapped today). No migration needed for existing M4-only deployments.

### Package layout

- **New:** `internal/lfs/locks/` — pure data layer.
  - `locks.go` — `Store` interface (`Create`, `List`, `Delete`, `Get`, `Verify`).
  - `sqlitestore.go` — sqlite implementation backed by the existing `*sqlitestore.Store` handle. Exposed as `New(db *sqlitestore.Store) *Store`.
  - `types.go` — `Lock`, `LockOwner`, `VerifyResult{Ours, Theirs}`.
  - `*_test.go` — unit tests against `:memory:` sqlite.

- **Extended:** `internal/lfs/handler.go` — register four new HTTP handlers, dispatched from new gateway routes.
- **Extended:** `internal/lfs/types.go` — wire-format request/response shapes (`LockRequest`, `LockResponse`, `ListLocksResponse`, `VerifyRequest`, `VerifyResponse`, `UnlockRequest`, `UnlockResponse`, `LockConflictResponse`).
- **Extended:** `internal/gateway/routes.go` — four new routes:
  - `POST /{t}/{r}.git/info/lfs/locks` (RequiredAction=ActionWrite, OpLFSLocksCreate)
  - `GET  /{t}/{r}.git/info/lfs/locks` (RequiredAction=ActionRead, OpLFSLocksList)
  - `POST /{t}/{r}.git/info/lfs/locks/verify` (RequiredAction=ActionRead, OpLFSLocksVerify)
  - `POST /{t}/{r}.git/info/lfs/locks/{id}/unlock` (RequiredAction=ActionWrite, OpLFSLocksUnlock)
- **Extended:** `internal/auth/sqlitestore/` — bootstrap the `lfs_locks` table in the schema-init code path. The `*sqlitestore.Store` handle gains no new methods; the locks package operates on the raw `*sql.DB` exposed by an existing accessor (or a new narrow `DB()` method if absent).
- **Extended:** `cmd/bucketvcs/serve.go` — pass the locks store into the LFS handler dependencies.

### Auth integration

The four lock handlers run after `gateway.RunAuth`, which validates the M4 token and attaches `*auth.Actor` to the request context. The LFS handler picks up the actor via the existing `deps.ActorFromContext` seam (same shape M13 batch uses).

- `Create` and `Unlock` require `auth.ActionWrite` on the repo. Gateway enforces this before the handler runs.
- `List` and `Verify` require `auth.ActionRead`. Verify returns ours/theirs by comparing each lock's `owner_user_id` to the caller's `Actor.UserID`.
- `force=true` on unlock requires the same `ActionWrite` permission the route already enforces; no special "admin" tier. The non-owner case is allowed because the caller already has write access (which is the spec-defined gate).

### Wire format

All endpoints accept and emit `application/vnd.git-lfs+json`. Per the LFS spec:

```
POST /info/lfs/locks
  → 201 LockResponse{ lock: Lock{id, path, locked_at, owner{name}} }
  → 409 LockConflictResponse{ lock: <existing>, message: "..." }
  → 403 / 401 / 404 per gateway auth path

GET /info/lfs/locks?path=<...>&id=<...>&cursor=<...>&limit=<N>&refspec=<...>
  → 200 ListLocksResponse{ locks: [...], next_cursor: "" }

POST /info/lfs/locks/verify
  body: { cursor?: "...", limit?: N, ref?: {name} }
  → 200 VerifyResponse{ ours: [...], theirs: [...], next_cursor: "" }

POST /info/lfs/locks/{id}/unlock
  body: { force: bool }
  → 200 UnlockResponse{ lock: <deleted lock> }
  → 403 if !force and caller != owner
  → 404 if not found
```

Pagination: implemented for List and Verify using `LIMIT ? OFFSET ?` against `id` ordering. Cursor is the next OFFSET serialized to a base16 string (opaque to clients). Default limit 100, max 1000.

### Lock ID format

`lock_` + 26 chars of base32-encoded random (160 bits). Generated by a new helper in `internal/lfs/locks/`. Avoids ULID's lexicographic-time leakage and matches the existing `bvts_` token-style naming. Globally unique by birthday-paradox argument; UNIQUE(id) at the DB layer is a safety net.

### Data flow — Lock create

```
client (git lfs lock)
  → POST /info/lfs/locks { path, ref: {name} }
gateway routes.go → ParseRoute → OpLFSLocksCreate, ActionWrite
gateway RunAuth → validates Bearer/Basic → Actor in ctx
LFS handler.LocksCreate:
  - decode body
  - actor := ActorFromContext(ctx); if nil, 401
  - generate id := generateLockID()
  - store.Create(ctx, Lock{id, tenant, repo, path, ref_name, owner_user_id=actor.UserID, locked_at=time.Now().Unix()})
    - sqlite INSERT; UNIQUE violation → return ErrAlreadyLocked, existing lock
  - handler maps:
    - success → 201 LockResponse with owner_name from users table lookup (or actor.UserName if cached)
    - ErrAlreadyLocked → 409 LockConflictResponse
    - other → 500
  - emit audit event lfs.lock.create
  - increment metric lfs_locks_created_total{outcome}
```

### Data flow — Verify

```
client (git lfs push)
  → POST /info/lfs/locks/verify { ref: {name}, cursor?, limit? }
LFS handler.LocksVerify:
  - actor := ActorFromContext(ctx); if nil, 401
  - locks, nextCursor := store.List(ctx, tenant, repo, ref_name?, cursor, limit)
  - partition by owner_user_id == actor.UserID
  - return { ours, theirs, next_cursor }
```

The ref filter respects null ref_name as "applies to all refs". If the request specifies `ref.name="refs/heads/release"`, the server returns locks where `ref_name="refs/heads/release"` OR `ref_name IS NULL`.

### Data flow — Unlock

```
client (git lfs unlock)
  → POST /info/lfs/locks/<id>/unlock { force }
LFS handler.LocksUnlock:
  - actor := ActorFromContext(ctx); if nil, 401
  - lock, err := store.Get(ctx, id); 404 if not found
  - verify lock.tenant + lock.repo match URL
  - if lock.owner_user_id != actor.UserID && !force: 403
  - store.Delete(ctx, id)
  - return UnlockResponse{lock}
  - emit audit event lfs.lock.delete{force, force_target_user_id=lock.owner_user_id}
```

The tenant+repo match check defends against a client guessing IDs from another repo — defense in depth on top of the existing per-repo gateway routing.

### Observability

Adds to existing M13 LFS metric/audit machinery in `internal/lfs/metrics.go` and `internal/lfs/audit.go`:

- **Metrics (4 new counters)**:
  - `lfs_locks_created_total{outcome}` — created / conflict / error
  - `lfs_locks_listed_total{outcome}` — success / error
  - `lfs_locks_verified_total{outcome}` — success / error
  - `lfs_locks_deleted_total{force, outcome}` — labels: `force=true|false`, `outcome=owner|forced|denied|not_found|error`
- **Audit events (3 new types)**:
  - `lfs.lock.create` — fields: `tenant, repo, lock_id, path, ref_name, owner_user_id`
  - `lfs.lock.delete` — fields: `tenant, repo, lock_id, force, force_target_user_id` (when force≠owner)
  - `lfs.lock.verify` — fields: `tenant, repo, ours_count, theirs_count` (no per-lock body — large lists would bloat audit logs)

List has no audit event (read-only enumeration, like Batch's read path).

### Failure modes

| Symptom | Cause | Action |
|---|---|---|
| 409 on lock create | UNIQUE(tenant, repo, path) violation | Client gets existing lock metadata; re-run with a different path |
| 403 on unlock without force | Caller is not the owner | Re-run with `force=true` if caller has write perms |
| 404 on unlock | Lock ID does not exist in this repo | Stale client state; run `git lfs locks` to refresh |
| 401 anywhere | Anonymous or invalid token | Send credentials |
| 500 from store | sqlite write failure (disk full, permissions) | Operator investigation |

## Testing

Three layers:

1. **Unit (`internal/lfs/locks/sqlitestore_test.go`)** — `Create / Get / List / Delete / Verify` against `:memory:` sqlite. Includes the UNIQUE-violation path, pagination cursor round-trip, ref-name filter, owner partition.

2. **Handler (`internal/lfs/handler_test.go` extensions)** — table-driven HTTP-level tests via `httptest.NewRecorder` for the four endpoints. Covers 201 / 409 / 403 / 404 / 401 status codes, wire-format JSON shape (per LFS spec), and the audit/metric side effects.

3. **End-to-end (`internal/gateway/e2e_locks_test.go`)** — boot the full gateway, run a real `git-lfs` CLI subprocess against it, exercise `git lfs lock`, `git lfs locks`, `git lfs unlock`, `git lfs unlock --force` (as a different user), and assert the server-side state matches each operation. Mirrors the existing M13 e2e LFS test pattern (`internal/lfs/e2e_test.go`).

Plus a smoke script `scripts/m13.3-lfs-locks-smoke.sh` exercising the same against localfs, on the M11/M12/M13 smoke pattern.

## Migration / compatibility

- **Schema:** the new `lfs_locks` table is created lazily on Open. Operators on M13 do nothing — restart the gateway and the table appears.
- **Backward compat:** pre-M13.3 binaries reading the same authdb file are unaffected (they don't see the new table). Pre-M13.3 LFS clients ignore the new endpoints since the server only advertises them if asked (LFS clients probe Locks via 404 detection).
- **No schema-version bump** on the authdb — adding a new table is additive and doesn't break the M4 schema contract.

## Deferred (explicit non-goals)

- TTL / auto-expiry on stale locks. Add later if real workflows show abandoned-lock pileup.
- SSH-native lock transport (`git-lfs-transfer` SSH protocol). Today's SSH-redirect-to-HTTPS pattern is consistent with how M13 handles upload/download via SSH.
- Lock import/export tooling. Operators query sqlite directly if needed.
- Lock notifications / webhooks. Out of scope for the data-plane milestone.
- Locking integration with M11 protected branches. Orthogonal — protected branches refuse pushes; locks block specific paths within an otherwise-pushable branch.

# M9.5 Bitmaps — design spec

**Status:** design draft (2026-05-19)
**Predecessor:** M9 (background maintenance, tag `m9-complete`)
**Worktree:** `.claude/worktrees/m95-bitmaps`, branch `m95-bitmaps`

## 1. Goal

Generate Git pack-bitmap (`.bitmap`) sidecars during background maintenance and reference them from the root manifest, so that real `git upload-pack` (running inside the lazy mirror) uses them to accelerate clone-time reachability walks.

Closes the explicit deferral from M9 ("bitmaps split off to M9.5") and the §15.3 "force repack when bitmap coverage falls below threshold" trigger that ships inert today.

## 2. Background

M9 introduced `bucketvcs maintenance`, which materializes the repo into a bare clone, runs `git pack-objects` via `internal/gitcli.PackObjectsAll`, and uploads the consolidated pack pair (`.pack` + `.idx`) to `packs/canonical/`. The result is recorded as a new `manifest.PackEntry`.

Git's pack-objects can also emit a `.bitmap` sidecar via `--write-bitmap-index`. The `.bitmap` is consumed by upload-pack for `clone` and broad `fetch` operations — Git's reachability walk reads the bitmap from the on-disk pack and short-circuits the per-object DAG traversal.

bucketvcs's pure-Go upload-pack negotiator (M10) does not consume `.bitmap` files. It uses `.bvrd` + `.bvcg` for reachability. However, the lazy-mirror path that backs the actual pack send invokes real `git upload-pack` against the materialized bare clone — which DOES read `.bitmap` when present. So the value chain is:

> maintenance writes `.bitmap` → upload to `packs/canonical/<id>.bitmap` → mirror materialize downloads `.bitmap` alongside `.pack`/`.idx` → real `git upload-pack` consumes it on clone.

Storage layout (per spec §6, §21) already accommodates this — `keys.Repo.PackBitmapKey(packHash)` returns the canonical path. The TODO sites in `internal/gc/discover.go:16` and `internal/gc/liveset.go:51` mark where M9.5 wires the new manifest field through GC.

## 3. Non-goals

- **No multi-pack index (`.midx`).** Spec phase 3 item. Bitmaps in M9.5 only apply to the single canonical pack each repack produces.
- **No pure-Go bitmap writer.** M9.5 reuses real `git pack-objects --write-bitmap-index`, same precedent as the M9 pack writer.
- **No bitmap consumer in bucketvcs's pure-Go upload-pack.** M10's pure-Go negotiator is unchanged. M9.5 is upstream-tooling acceleration only.
- **No retroactive bitmap generation for non-maintenance pack writers.** receive-pack does NOT emit bitmaps (small, recent, one-bitmap-per-push would be churny). Bitmaps live exclusively on canonical packs produced by maintenance.

## 4. Architecture

### 4.1 Manifest schema

`manifest.PackEntry` gains one optional field:

```go
type PackEntry struct {
    PackID       string `json:"pack_id"`
    PackKey      string `json:"pack_key"`
    IdxKey       string `json:"idx_key"`
    SizeBytes    int64  `json:"size_bytes"`
    ObjectCount  int    `json:"object_count"`
    PackChecksum string `json:"pack_checksum,omitempty"`

    // BitmapKey is the object-store key of the .bitmap sidecar for
    // this pack. Set when maintenance produced a bitmap; empty for
    // packs written by receive-pack (which never emits bitmaps) or
    // pre-M9.5 maintenance runs.
    BitmapKey string `json:"bitmap_key,omitempty"`
}
```

**Schema version:** unchanged at 1. The field is `omitempty` and absence is valid (per spec §35 GC safety: "an unrecognized bitmap field is treated as 'no bitmap'"). `SupportedReaderVersion` is not bumped.

### 4.2 gitcli surface

`gitcli.PackObjectsAll` gains a sibling function rather than a variadic option, to keep the M9 callsite stable and the M9.5 callsite explicit:

```go
// PackObjectsAllWithBitmap is identical to PackObjectsAll plus
// --write-bitmap-index passed to git pack-objects. The output paths
// are <prefix>-<id>.pack, .idx, and .bitmap. Returns the pack ID; the
// caller derives bitmap path as prefix-id.bitmap.
//
// pack-objects writes the bitmap only when it has full reachability
// for the input object set, which is exactly the case when the input
// is `rev-list --all --objects` (M9's pattern). Empty repos still
// fail the same way they do in PackObjectsAll.
func PackObjectsAllWithBitmap(ctx context.Context, dir, outPrefix string) (string, error)
```

Rationale for the sibling rather than the option: PackObjectsAll has 5 call sites today (importer, exporter, maintenance, tests). Most of them should NOT emit bitmaps (importer/exporter are scratch; tests are unit-level). Making the bitmap explicit at the callsite is safer than threading a bool through 5 places.

### 4.3 Maintenance pipeline change

The repack phase changes call from `PackObjectsAll` to `PackObjectsAllWithBitmap`. `RepackOutput` gains:

```go
type RepackOutput struct {
    // ...existing fields...
    BitmapPath string // <bareDir>/out/pack-<PackID>.bitmap (may not exist if pack-objects skipped it)
}
```

`uploadInput` gains an optional `BitmapBytes`/`BitmapPath`. `uploadArtifacts` PutIfAbsent's the bitmap to `keys.Repo.PackBitmapKey(packID)` when present. `uploadResult` gains `BitmapKey`. The CAS-merge in pipeline.go records `PackEntry.BitmapKey = uploaded.BitmapKey`.

**Compact-only path:** unchanged. Compact-only does not rewrite the pack, so the existing `PackEntry.BitmapKey` (if any) is preserved through the CAS body builder.

### 4.4 GC integration

Two existing TODO sites are wired:

- `internal/gc/liveset.go:51` — when iterating `body.Packs`, add `BitmapKey` (if non-empty) to the live key set. Otherwise an existing `.bitmap` would be swept on the next GC cycle.
- `internal/gc/discover.go:16` — `.bitmap` files become candidates for sweep. Any `.bitmap` under `packs/canonical/` not in the live set is unreachable.

Test additions follow the existing pattern: synthesize a manifest with `BitmapKey`, run the liveset/discover round trip, assert the bitmap survives one cycle and is swept after manifest reference removal.

### 4.5 Coverage trigger (§15.3 #5)

Adds two fields to `Thresholds`:

```go
type Thresholds struct {
    // ...existing...

    // BitmapCoveragePct triggers when fewer than this percent of
    // canonical packs (by count, not bytes) carry a non-empty
    // BitmapKey. Range [0, 100]. 0 disables the trigger entirely
    // (matches M9/M10 convention).
    BitmapCoveragePct int
}
```

`DefaultThresholds` sets `BitmapCoveragePct = 100` — any missing bitmap forces a repack. Operators can dial this down for repos where post-repack `.bitmap` generation occasionally fails (rare; `pack-objects --write-bitmap-index` is reliable when the input set is `rev-list --all`).

`TriggerReport` gains `BitmapCoveragePct int` (the computed coverage) + the existing `Reason` string surfaces `"bitmap_coverage:<pct>%"`.

Evaluation is cheap (pure manifest scan), so it sits in `evaluatePure` alongside the other cheap triggers — no new IO cost in Phase 0.

### 4.6 Operator guide

`docs/m9-maintenance-operator-guide.md` gains:
- A subsection under §15.3 documenting the new `--bitmap-coverage-pct` flag and what `bitmap_coverage:N%` means in trigger reports.
- A one-paragraph mention in the storage-layout section explaining what `.bitmap` files are, who writes them, who reads them, and why receive-pack doesn't emit them.

### 4.7 Smoke / regression coverage

No new smoke script. The existing `scripts/m9-maintenance-smoke.sh` (if present) gains an assertion that a `.bitmap` lands at the expected key after a maintenance run. The existing `m13-lfs-smoke-*.sh` are unaffected (they don't exercise maintenance).

The differential harness already round-trips full maintenance through import/export; no changes needed there — the test repos are small enough that pack-objects will emit a `.bitmap` for them, and the round-trip stays byte-identical.

## 5. Migration

- **Pre-M9.5 manifests** load with `BitmapKey == ""` everywhere. The coverage trigger fires on the next maintenance run, which repacks and writes bitmaps for every canonical pack. After one cycle the manifest is fully bitmapped.
- **Operators who roll back** to a pre-M9.5 binary continue to work — the unknown `bitmap_key` JSON field is ignored on read (encoding/json default). GC under pre-M9.5 won't see the new live key, so the `.bitmap` files will be swept on the first GC cycle. Acceptable: the next M9.5 maintenance run regenerates them.
- **No data migration.** No backfill job. The trigger does the work organically.
- **Pack-id churn on first M9.5 run.** Because M9.5 switches `maintenance.Repack` from `gitcli.PackObjectsAll` (rev-list piped into pack-objects --stdin) to `gitcli.PackObjectsAllWithBitmap` (pack-objects --revs --all directly), the first maintenance run after upgrade will produce a new pack-id / PackChecksum for every repo, even on identical input — the two invocations cover the same object set but choose different delta encodings. Downstream consumers that pin pack-ids (reproducibility harnesses, diff tools across milestones) should refresh after upgrade. The old packs become unreachable via the manifest swap and are GC-collected after retention.

## 6. Risk surface

- **`pack-objects --write-bitmap-index` silently produces no bitmap when the pack lacks a HEAD.** Maintenance always materializes from a manifest that has refs, so this should be a non-issue, but the implementation MUST tolerate a missing bitmap output file (treat as "no bitmap written this run"; do not error out).
- **Storage-key clash with pre-existing keys.** None — `.bitmap` keys are under `packs/canonical/<id>.bitmap` and packs are content-addressed by their SHA-1 trailer. A re-uploaded canonical pack with an identical SHA-1 has byte-identical contents; `PutIfAbsent` of the `.bitmap` either no-ops or wins.
- **Cost of one extra PUT per maintenance run.** Bitmaps for typical repos are 1–10% of pack size. One additional small PUT per maintenance is negligible relative to the pack PUT itself.
- **`PackObjectsAllWithBitmap` failure mode different from `PackObjectsAll`.** Tests must cover the case where `--write-bitmap-index` causes pack-objects to fail (it shouldn't, but the call needs to surface the error cleanly, not swallow it).

## 7. Phase breakdown

P1 — Manifest schema + GC integration
- Add `PackEntry.BitmapKey` field with omitempty tag.
- Wire `internal/gc/liveset.go` and `internal/gc/discover.go` TODO sites.
- Unit tests for liveset/discover with `BitmapKey` set.
- Manifest schema test asserting the new field round-trips.

P2 — gitcli surface
- Add `PackObjectsAllWithBitmap` alongside `PackObjectsAll`.
- Unit tests: bitmap file is produced; pack ID matches the no-bitmap path's output for the same input (sanity: only the sidecar differs).
- Empty-repo case still errors the same way.

P3 — Maintenance pipeline wiring
- `RepackOutput.BitmapPath` populated when present.
- `uploadInput`/`uploadResult` carry bitmap; `uploadArtifacts` PutIfAbsent's the bitmap.
- CAS-merge body builder writes `PackEntry.BitmapKey`.
- Pipeline integration test: full maintenance run on a small fixture produces a manifest with `BitmapKey` populated and the bitmap blob at the expected storage key.

P4 — Coverage trigger
- `Thresholds.BitmapCoveragePct` field + `DefaultThresholds` setting it to 100.
- `evaluatePure` computes coverage and sets `TriggerReport.Reason = "bitmap_coverage:N%"` when below threshold.
- `RunOptions` / CLI flag `--bitmap-coverage-pct` for operator override.
- Unit tests for the trigger evaluation matrix.

P5 — Operator guide + smoke regression + squash + tag
- Operator guide updates (storage layout + §15.3 subsection + CLI flag table).
- Run existing maintenance smoke (if any) + full `go test ./...`.
- Spec + code-quality review per M1+ protocol.
- roborev-refine.
- Squash to main, tag `m9.5-complete`.
- Memory entries: new `m95_progress.md`, MEMORY.md row, m9_progress.md "Closed in M9.5" note.

## 8. Out of scope

- Multi-pack index `.midx` (spec phase 3).
- Bitmap consumption in the pure-Go upload-pack negotiator.
- Bitmaps on receive-pack-written packs.
- Reverse-index (`.rev`) sidecars. Git 2.31+ emits these alongside `.bitmap`; they accelerate object-lookup-by-offset. Worth a follow-up phase but not load-bearing for clone perf.
- Per-tenant bitmap quotas / observability beyond the existing maintenance metrics.

## 9. Open questions

None at design time. The architecture is mechanical and the prior art (M9 pack production, M11 bundle production) is well-established.

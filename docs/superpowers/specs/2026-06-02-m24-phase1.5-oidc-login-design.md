# M24 Web UI — Phase 1.5: OIDC Browser Login

**Status:** Design approved (2026-06-02)
**Builds on:** M24 Phase 1 (web UI chassis + identity, commit `6c02bbd`) and M22 OIDC token-exchange (`internal/oidc`).
**Scope:** Add OAuth2 authorization-code + PKCE **relying-party** login to the web UI so browser users can sign in via a single external OIDC provider. Identity links to a **pre-provisioned** local user via verified-email trust-on-first-use. No auto-provisioning, no multi-IdP, no code/admin UI (those are Phases 2/3).

---

## 1. Background & decisions

M24 Phase 1 shipped password login over cookie sessions and *reserved* OIDC as a fast-follow. The session layer already records a `provider` string (`"password"` today), so OIDC slots in as new handlers that end in the same `CreateSession(…, "oidc", …)` + cookie + redirect — **no provider-interface refactor** (there is no `Provider` interface in `internal/web`; password login is inlined).

### Locked decisions (from brainstorming)

| Decision | Choice |
|---|---|
| RP implementation | **Reuse `internal/oidc.Verifier`** (discovery/JWKS/signature/`iss`/`exp`) + **`golang.org/x/oauth2`** for the code exchange; hand-write the small `aud` + `nonce` checks |
| Account linking | **Verified-email trust-on-first-use (TOFU):** resolve by `(issuer, subject)`, else match a pre-provisioned user by verified email and **pin** `(issuer, subject)`. Reject unknown / unverified / disabled. No auto-provisioning |
| IdP count | **Single** configured provider (serve flags) |
| Email storage | **Add `users.email`** (nullable) — usernames are not emails (`alice`, `bob`), so overloading `name` is wrong |
| Client secret | Optional (confidential client *or* public client + PKCE); supplied via **file or env var, never argv** |
| Coexistence | Password form stays; an SSO link is added when OIDC is enabled |

## 2. Architecture

### 2.1 Reuse surface (verified against `main`)

| Reused | Location | Role in RP flow |
|---|---|---|
| `oidc.NewVerifier()` / `Verifier.Verify(ctx, raw, issuer) (Claims, error)` | `internal/oidc/verify.go` | ID-token signature + `iss`/`exp`/`nbf`/`iat` (rejects `none`/HMAC; RS/ES only). Discovery + JWKS cached |

| `golang.org/x/oauth2` v0.36.0 | go.mod (indirect → promote to direct) | Authorization URL + code→token exchange + PKCE |

`internal/oidc.Verify` does **not** check `aud` or `nonce` (RP-specific) — the callback adds those.

## 3. Schema — migration 0014

> Next number confirmed against the tree: latest is `0013`.

`users.email` is nullable and **case-insensitively unique when present** (enforced in code via lower-casing on write + a unique index on the stored value, which is always stored lower-cased). `user_identities.issuer` stores the **literal issuer URL** (not a FK to `oidc_issuers`) to keep browser login decoupled from M22's token-exchange registry.

### Store methods (`internal/auth/sqlitestore/identities.go`)
```go
SetEmail(ctx, userName, email string) error               // lower-cases; ErrNoSuchUser; ErrConflict on dup
FindUserByEmail(ctx, email string) (*auth.Actor, error)   // lower-cased lookup; ErrNoSuchUser; honors disabled? (see §4.7)
FindIdentity(ctx, issuer, subject string) (*auth.Actor, error) // joins users; ErrNoSuchUser if unlinked
LinkIdentity(ctx, userID, issuer, subject, email string) error // INSERT; ErrConflict if (issuer,subject) taken
ListIdentities(ctx, userName string) ([]Identity, error)
RemoveIdentity(ctx, issuer, subject string) error
```
`FindUserByEmail`/`FindIdentity` return an `*auth.Actor` (`{UserID, Name, IsAdmin, Scopes: ScopeLegacy}`) so the web layer never imports the storage `User`/`Identity` types — same decoupling as M24.

## 4. The RP flow

### 4.1 `GET /login/oidc` (authorize)
1. If OIDC not configured → `404` (route not registered).
2. Generate: `state` (32 random bytes, base64url), `nonce` (32 random bytes), PKCE `code_verifier` (43–128 chars) + `code_challenge = BASE64URL(SHA256(verifier))`.
3. Write a short-lived (**10 min**) HttpOnly cookie `bvcs_oidc` carrying an **HMAC-authenticated** (per-process key) encoding of `{state, nonce, code_verifier, next}` (`next` = `safeNext(?next=)`) so its contents cannot be forged. Flags: HttpOnly, SameSite=Lax, Secure-when-TLS, `Path=/login/oidc`.
4. Redirect (`302`) to `oauth2.Config.AuthCodeURL(state, oidc.Nonce(nonce), PKCE S256 params)` — scopes default `openid email profile`.

### 4.2 `GET /login/oidc/callback`
Validation order (fail-closed; every rejection emits `auth.oidc.rejected` with a `reason` and increments the M18 limiter keyed by client IP):
1. Read `bvcs_oidc` cookie; missing/expired/undecodable → reject `state_mismatch`.
2. If query has `error` → render the IdP error (reason `idp_error`).
3. Query `state` must equal cookie `state` (constant-time) → else `state_mismatch`.
4. Exchange `code` (+ `code_verifier`) at the token endpoint via `oauth2.Config.Exchange` → extract `id_token` from the token `Extra("id_token")`. Missing → `token_invalid`.
5. `oidc.Verifier.Verify(ctx, idToken, issuerURL)` → `Claims` (signature, `iss`, `exp`, `nbf`, `iat`). Error → `token_invalid`.
6. RP checks: `aud` contains `client_id` (string or array form); `nonce` claim equals cookie `nonce` (constant-time). Fail → `token_invalid`.
7. Require `email_verified == true` and non-empty `email`. Fail → `email_unverified`.
8. Resolve user: `FindIdentity(issuer, sub)`. If found → that actor. Else `FindUserByEmail(email)`:
   - found & enabled → `LinkIdentity(userID, issuer, sub, email)` (emit `auth.oidc.identity_linked`), use that actor;
   - not found → `no_user`; on `LinkIdentity` `ErrConflict` (race) → re-`FindIdentity` and proceed.
9. Disabled user at any point → `disabled`.
10. `CreateSession(actor.UserID, "oidc", ttl)`; set `bvcs_session` cookie (HttpOnly, SameSite=Lax, Secure-when-TLS); delete `bvcs_oidc`; emit `auth.oidc.login` + `web_login_total{provider=oidc,result=success}`; `303` redirect to `safeNext(next)`.

### 4.3 Security properties
- **CSRF:** the `state` double-submit (cookie vs query). The temp cookie is HttpOnly + short-lived; an attacker cannot both set our cookie and obtain a validly-signed ID token whose `nonce` matches.
- **Replay:** `nonce` binds the ID token to this login attempt.
- **Code interception:** PKCE S256.
- **Algorithm confusion:** inherited from `internal/oidc` (asymmetric only; `none`/HMAC rejected).
- **No open redirect:** `next` passes through `safeNext` (rejects `//`, `\`, absolute).
- **No secrets logged:** audits carry `issuer`, `subject` (opaque), `email`, `reason` — never tokens or the client secret.

### 4.4 Config (serve flags)
Distinct from M22's token-exchange `--oidc` (verifies inbound tokens; this is an OAuth client).

| Flag / env | Purpose |
|---|---|
| `--oidc-login` (bool, default false) | Enable browser RP login |
| `--oidc-login-issuer` | Issuer URL, e.g. `https://accounts.google.com` |
| `--oidc-login-client-id` | OAuth2 client id |
| `--oidc-login-client-secret-file` **or** `BUCKETVCS_OIDC_LOGIN_CLIENT_SECRET` | Client secret (optional; never on argv) |
| `--oidc-login-redirect-url` | Registered callback, e.g. `https://host/login/oidc/callback` |
| `--oidc-login-scopes` (default `openid,email,profile`) | Requested scopes |
| `--oidc-login-label` (default `Single sign-on`) | Login-page button text |

**Startup validation:** `--oidc-login=true` hard-requires `issuer`, `client-id`, and `redirect-url` (fail-fast, exit 2 with a clear message). The verifier (`oidc.NewVerifier()`) and an `*oauth2.Config` are built once and passed into `web.Deps`. OIDC login requires an HTTP listener (same block as the rest of the UI); if `--oidc-login` is set without `--addr`, emit the existing M24 startup WARN.

### 4.5 Login page
When OIDC is enabled, `login.html` shows the password form **and** a link/button to `/login/oidc` labeled with `--oidc-login-label`. When disabled, the page is unchanged. `?next=` is preserved through the SSO link.

### 4.6 CLI
- `bucketvcs user add <name> [--email <e>] [--admin]` — `--email` added (lower-cased; `ErrConflict` on dup).
- `bucketvcs user set-email <name> <email>` — set/clear (empty clears).
- `bucketvcs user identity list <name>` — NDJSON of linked identities.
- `bucketvcs user identity remove <issuer> <subject>` — unlink (e.g. subject reassigned).

### 4.7 Disabled-user handling
`FindUserByEmail` and `FindIdentity` return the actor **regardless of disabled state**; the callback checks `disabled` explicitly and rejects with `disabled` (parallels the password path, where a correct credential on a disabled account returns `ErrUserDisabled`). Disabled users therefore cannot get a session but the distinct reason is auditable.

## 5. Observability

- **Metric:** `web_login_total{provider,result}` — `provider ∈ password|oidc`, `result ∈ success|invalid|ratelimited` (the password path passes `provider=password`; this is a backward-compatible label addition).
- **Audits:** `auth.oidc.login{user_id,user,issuer,subject}`, `auth.oidc.identity_linked{user_id,user,issuer,subject,email}`, `auth.oidc.rejected{issuer,reason,email?}`.

## 6. Testing

- **Store:** migration 0014 schema present; `SetEmail` (lower-case, dup→ErrConflict, unknown→ErrNoSuchUser); `FindUserByEmail` (case-insensitive, unknown→ErrNoSuchUser); `LinkIdentity` + `FindIdentity` round-trip; `(issuer,subject)` uniqueness → ErrConflict; `RemoveIdentity`; cascade on user delete.
- **State cookie:** `bvcs_oidc` encode/decode round-trip; tamper/expiry → reject.
- **Callback (table-driven, with a fake verifier + fake token endpoint):** happy path (TOFU link → session); pre-linked identity (no re-link); `state` mismatch; missing/expired temp cookie; `nonce` mismatch; `aud` mismatch; `email_verified=false`; unknown email (`no_user`); disabled user; IdP `error=` param; malformed/expired ID token. Each asserts status, no session cookie on failure, and the audit `reason`.
- **Authorize:** sets `bvcs_oidc` + redirects to a URL containing `client_id`, `state`, `nonce`, `code_challenge`, `code_challenge_method=S256`, scopes.
- **Open redirect:** `next` through `safeNext` on the OIDC path.
- **CLI:** `user add --email`, `set-email`, `identity list/remove` exit codes + effects.
- **serve:** `--oidc-login` without required flags → exit 2; enabled → `/login/oidc` registered and redirects (302) to the issuer's authorization endpoint (use a local fake issuer with a discovery doc).
- **Smoke** (`scripts/oidc-login-smoke-localfs.sh`): a stub IdP (static discovery + JWKS + a signed ID token) driving the full authorize→callback→session→landing flow on localfs.

## 7. Out of scope (deferred)
Multiple IdPs; auto-provisioning of unknown users; RP-initiated logout (`end_session_endpoint`); group/role claim → admin mapping; OIDC-only mode (hiding the password form); refresh-token storage / silent renewal; account-merge UI; per-tenant IdPs.

## 8. Risks & mitigations
| Risk | Mitigation |
|---|---|
| Hand-rolled `aud`/`nonce` checks wrong | Constant-time compares; table-driven negative tests for each; reuse audited `internal/oidc` for the hard parts (sig/discovery/JWKS) |
| Email-based TOFU links the wrong user | Require `email_verified=true`; unique `users.email`; pin `(issuer,subject)` after first link so email changes don't re-link |
| Temp-cookie tampering | HttpOnly + short TTL + double-submit `state`; signed/authenticated encoding of the `bvcs_oidc` blob (HMAC with a per-process key) so contents can't be forged |
| Client secret leakage | File/env only, never argv; never logged |
| Migration numbering drift | Verified next = **0014** against the tree |
| Decoupling regression | `internal/web` imports only `auth` + `DataStore`; new methods returned as `*auth.Actor`, not storage types |

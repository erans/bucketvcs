# M13.4 — LFS GC design

**Status:** draft (under review)
**Date:** 2026-05-20
**Predecessor:** M13.3 (LFS Locks, commit 15e0853)
**Tag target:** `m13.4-lfs-gc`

## Goal

Implement mark-and-sweep garbage collection for Git LFS objects. Closes operator-guide §8.3 deferred item. After this milestone:

- `bucketvcs gc --lfs --repo=<t>/<r>` discovers unreferenced LFS objects and (after retention) deletes them.
- `--all-repos` extends across every repo, like the existing Git-objects GC.
- `--mark-only`, `--sweep-only`, `--dry-run` work as for Git GC.
- The discovery walk inspects every reachable Git blob, identifies LFS pointer blobs, extracts the pointed-to LFS object OID, and builds a live set. No new persistent state at push time.
- Retention is time-based (default 7 days), matching M8.
- Operationally fail-soft: a wrongly-deleted LFS object is transparently re-PUT by the next client push (LFS PUT is content-addressed + idempotent).

Out of scope for M13.4: cross-repo pooling/dedupe, LFS-aware bandwidth metering, quotas (depends on this milestone), TTL on LFS objects, anything that touches the LFS object-store key layout.

## Architecture

### New package `internal/lfs/gc/`

Parallel to existing `internal/gc/` (Git-objects GC). Don't share code — the storage prefix, the live-set builder, and the mark record shape are all different. Sharing only invites coupling without gain.

```
internal/lfs/gc/
├── doc.go         — package-level overview
├── gc.go          — RunMark, RunSweep, top-level options + reports
├── liveset.go     — BuildLiveSet: walk mirror, find LFS pointers
├── marks.go       — mark-record persistence under .gc/lfs-marks/
├── sweep.go       — sweep logic + retention math
├── pointer.go     — LFS pointer file parser (small helper)
└── *_test.go      — unit tests, localfs integration tests
```

### CLI shape

`cmd/bucketvcs/gc.go` gains:

- `--lfs` (bool, default false) — enable LFS GC.
- All existing flags (`--repo`, `--all-repos`, `--mark-only`, `--sweep-only`, `--dry-run`, `--format`, `--store`) apply.
- Default behavior (no `--lfs`): unchanged — Git-objects GC only.
- `--lfs` alone: LFS GC only.
- `--lfs --include-git-objects`: both phases, sequentially (Git first, then LFS). This combined invocation makes the two phases observable in one run for ops who want a single cron entry.

The existing `RunReport` gains an optional `LFS *LFSReport` field set when `--lfs` ran.

### Storage layout (no new prefixes besides marks)

- LFS objects: existing `tenants/<t>/repos/<r>/lfs/objects/<sha256-hex>` (unchanged).
- LFS marks: NEW `tenants/<t>/repos/<r>/.gc/lfs-marks/<mark-id>.json`. Parallel to existing `.gc/marks/` (Git-objects marks); same directory ancestor but separate filenames so each kind of GC reads/writes its own.

Mark record shape:

```json
{
  "id": "lfs-20260520T120000Z-abcd",
  "started_at": "2026-05-20T12:00:00Z",
  "manifest_version_at_mark": 42,
  "candidates": [
    { "oid": "<sha256-hex>", "size_bytes": 12345, "first_seen_unreferenced_at": "2026-05-20T12:00:00Z" }
  ],
  "previous_mark_id": "lfs-20260513T120000Z-xyz"
}
```

`first_seen_unreferenced_at` is carried forward from the previous mark when an object remains unreferenced across runs. New orphans get the current `started_at` value. This preserves the retention clock across mark runs.

## Components

### `lfs/gc/pointer.go` — LFS pointer parser

A 30-line helper. Pure function:

```go
// ParsePointer inspects up to 200 bytes of a blob's contents and
// returns the referenced LFS object OID (hex sha256) if the blob
// is an LFS pointer file. Returns ("", false) if it's not a pointer.
//
// LFS pointer file format (LFS spec):
//   version https://git-lfs.github.com/spec/v1
//   oid sha256:<64-hex>
//   size <int>
//
// We only need the oid line; size is informational. 200 bytes is
// well above the maximum pointer file size in practice.
func ParsePointer(b []byte) (oid string, ok bool)
```

Tested with: valid pointer; missing oid line; wrong version; non-pointer (random binary). No deps beyond stdlib.

### `lfs/gc/liveset.go` — BuildLiveSet

```go
// BuildLiveSet enumerates every LFS object OID referenced by any
// reachable commit in the repo's materialized mirror. Returns the
// set of referenced OIDs (hex sha256, lowercase) for the sweep to
// preserve.
//
// Materializes the mirror via maintenance.Materialize if not already
// present at bareDir. Then:
//   1. git rev-list --objects --all → reachable (oid, path) pairs.
//   2. cat-file --batch-check '%(objectname) %(objecttype) %(objectsize)'
//      to filter to blob OIDs with size ≤ 1024 (LFS pointers cap at
//      ~200 bytes; 1024 is conservative).
//   3. cat-file --batch '%(objectname) %(objecttype) %(objectsize)\n'
//      for the candidate blobs; pipe contents through ParsePointer.
//   4. Union pointed-to LFS OIDs into the set.
//
// Cost: O(reachable blobs). For a typical repo this is bounded by
// the blob count (not byte count); --batch amortizes per-call git
// overhead.
func BuildLiveSet(ctx context.Context, bareDir string) (map[string]struct{}, error)
```

Tested with: a real bare repo (created via gitcli helpers) containing a mix of regular blobs and LFS pointer blobs; assert the returned set contains the pointer OIDs and only those.

### `lfs/gc/marks.go` — mark records

Mirrors `internal/gc/marks/` API surface (`Record`, `Write`, `ReadLatest`, etc.) but writes under `.gc/lfs-marks/` and carries the LFS-specific candidate shape. Don't share the type — the candidate fields differ (LFS marks carry size_bytes; Git marks carry pack keys).

### `lfs/gc/gc.go` — RunMark + RunSweep

```go
// MarkOptions / MarkReport / SweepOptions / SweepReport
type MarkOptions struct {
    Now              func() time.Time
    RetentionSeconds int
    DryRun           bool // future: a Mark dry-run is a no-op since marks
                          // are observation only. Currently ignored.
}

func RunMark(ctx context.Context, store storage.ObjectStore, r *repo.Repo, opts MarkOptions) (Record, error)
func RunSweep(ctx context.Context, store storage.ObjectStore, r *repo.Repo, mark Record, opts SweepOptions) (SweepReport, error)
```

Mark flow:
1. Materialize the mirror (use existing `maintenance.Materialize`).
2. `BuildLiveSet(ctx, bareDir)` → referenced OIDs.
3. `store.List(ctx, RepoLFSPrefix(tenant, repo))` → all LFS objects with size + key.
4. Diff: any LFS object NOT in liveset is a candidate for sweep.
5. Read latest prior mark; for each candidate already in the prior mark, carry forward `first_seen_unreferenced_at`. Otherwise stamp `now`.
6. Write new mark via `marks.Write`.

Sweep flow:
1. Read the mark record specified by caller (or latest).
2. For each candidate, check `now - firstSeenUnreferencedAt > retentionSeconds`.
3. If yes, `store.Delete(ctx, key)`. Aggregate counts + bytes deleted.
4. If `DryRun`, skip the Delete; report what would have been deleted.
5. Write sweep record under `.gc/lfs-sweeps/<id>.json` (mirror M8 pattern).
6. Return `SweepReport{Objects: N, Bytes: M, Skipped: K, Errors: ...}`.

### `cmd/bucketvcs/gc.go` — wiring

Add `--lfs` and `--include-git-objects` bool flags. Dispatch:

```go
runLFS := *lfs
runGit := !*lfs || *includeGitObjects

if runGit { /* existing path */ }
if runLFS {
    lfsMark, _ := lfsGC.RunMark(...)
    if !*sweepOnly { /* write mark */ }
    if !*markOnly  { lfsGC.RunSweep(...) }
}
```

The default (no `--lfs`) is unchanged. Mutually-exclusive flag validation already exists for `--mark-only` vs `--sweep-only`.

Per-phase RunReport additions:
```go
type RunReport struct {
    // existing fields ...
    LFS *LFSReport `json:"lfs,omitempty"`
}
```

## Data flow — example

A 3-commit repo: commit A adds `model.bin` (LFS), commit B updates it, commit C deletes it. Refs: `main` points at C.

Pre-GC storage:
- `lfs/objects/<sha-of-A's-model>` (referenced by A's tree only)
- `lfs/objects/<sha-of-B's-model>` (referenced by B's tree only)

C's tree references neither — `model.bin` is deleted from C's tree.

Mark:
- `BuildLiveSet` walks `--all` refs, sees commits A, B, C reachable from `main`.
- A's tree has a pointer to `<sha-of-A>`; B's tree has a pointer to `<sha-of-B>`.
- Liveset = {`<sha-of-A>`, `<sha-of-B>`}.
- Storage list returns both objects.
- Diff: empty (both are in liveset).
- Mark records 0 candidates.

Now garbage-collect commits A and B by repacking + dropping their refs (e.g., via a force-push that rewrites `main` to a fresh commit with no `model.bin` history). After repack:
- `BuildLiveSet` walks only the new `main`; no pointer blobs found.
- Liveset = {}.
- Storage list returns both objects.
- Diff: both are candidates. Stamped with `now`.

Sweep, 7 days later (retention satisfied):
- For each candidate, retention check passes → delete.
- Report: `{Objects: 2, Bytes: ..., Errors: nil}`.

Sweep run immediately (before retention):
- Retention check fails → skip. Report: `{Objects: 0, Skipped: 2, ...}`.

## Push-race safety

Per the brainstorming, time-based retention only. Default 7 days. Justification documented in operator guide §8.3:

> LFS GC is fail-soft. Because LFS objects are content-addressed by SHA-256 and the LFS PUT path is idempotent, a wrongly-deleted object is transparently restored by the next client push: the `git lfs push` verify endpoint detects the missing object and the client re-uploads. No data loss; one round-trip cost. The 7-day retention default makes this race practically impossible in normal operation; a shorter retention may be appropriate for cost-sensitive deployments.

No tx-commit marker, no manifest-version check during sweep.

## Failure modes

| Symptom | Cause | Action |
|---|---|---|
| `BuildLiveSet: mirror materialize failed` | underlying maintenance.Materialize error (storage IO, git error) | Investigate logs; retry GC later |
| `BuildLiveSet: rev-list failed` | git binary error or repo corruption | Investigate; possibly re-mirror |
| `RunSweep: storage.Delete failed for <oid>: <err>` | per-object delete failure (network, permissions) | Logged + counted in report; sweep continues; retry next run |
| `lfs gc sweep: mark version <X> not found` | --sweep-only with a stale mark ID; the latest mark is newer | Re-run with --mark-only first, or omit --sweep-only |

## Observability

Adds to existing M13 LFS metric/audit machinery:

**Metrics (3 new counters)**:
- `lfs_gc_objects_marked_total{outcome}` — outcome ∈ {candidate, error}
- `lfs_gc_objects_swept_total{outcome}` — outcome ∈ {deleted, skipped_retention, error}
- `lfs_gc_bytes_swept_total` — total bytes freed (cumulative gauge / counter)

**Audit events (2 new types)**:
- `lfs.gc.mark` — fields: tenant, repo, mark_id, candidates_count, manifest_version
- `lfs.gc.sweep` — fields: tenant, repo, mark_id, sweep_id, deleted_count, deleted_bytes, skipped_count, error_count

No per-object audit (would balloon log volume on large sweeps). Operators investigating a specific deletion grep the structured logs at `Info`/`Warn`.

## Testing

Three layers:

1. **Unit**:
   - `pointer_test.go`: ParsePointer happy + 4 negative cases.
   - `liveset_test.go`: BuildLiveSet against a synthetic bare repo (created via `gitcli` helpers) containing 3 commits with various blob shapes.
   - `marks_test.go`: Write/ReadLatest round-trip + carry-forward of `first_seen_unreferenced_at`.

2. **Integration (`gc_test.go`)**:
   - Localfs: seed a repo + a known set of LFS objects via the LFS Batch path (or directly via store.Put); run mark + sweep; assert candidates + post-sweep state.
   - Retention-honored test: mark, advance fake clock by less than retention, sweep, assert no deletes.
   - Retention-expired test: mark, advance fake clock past retention, sweep, assert deletes.
   - Carry-forward test: mark twice with the same orphan; assert the first_seen timestamp survives.

3. **End-to-end (`scripts/m13.4-lfs-gc-smoke.sh`)**:
   - Init + serve + push a real LFS object via git-lfs.
   - Force-delete the referencing commit (orphan the object).
   - Run `bucketvcs gc --lfs --repo=... --retention=0s --include-git-objects` (zero retention for the smoke).
   - Assert the LFS object key is gone from storage.

Plus a `dry-run` assertion in the smoke that verifies nothing is deleted when `--dry-run` is passed.

## Operator guide

`docs/m13-lfs-operator-guide.md` §8.3 status flips from `❌ deferred` to `✅ implemented` and gains:

- A "how it works" sub-section: mark walks the mirror, sweep applies retention.
- A "cost" caveat: BuildLiveSet is O(reachable blobs); for very large repos this is slower than Git-objects GC. Suggests running LFS GC on a weekly cadence (vs the existing Git GC which can run daily).
- A "fail-soft" callout: the LFS PUT-is-idempotent property that justifies the time-based retention model.
- An updated metrics + audit table for the 3 new metrics + 2 new audit events.

## Migration / compatibility

- No schema bump. No manifest change.
- First GC run after upgrade creates the `.gc/lfs-marks/` prefix lazily. No migration step.
- Backward compatible: pre-M13.4 binaries running `bucketvcs gc` continue to work (no `--lfs` flag = unchanged behavior).
- Forward compatible: a future M13.5 (quotas) can read the LFS mark records to compute "freed bytes per tenant" without schema changes.

## Deferred (explicit non-goals)

- **Cross-repo dedupe**: LFS objects stay per-repo (M13 spec §4). Not changed by M13.4.
- **Bandwidth metering** (operator-guide §8.5): orthogonal.
- **Quotas** (operator-guide §8.4): blocked on this milestone; ships in a separate M13.5.
- **TTL on LFS objects** (auto-expiry of LFS data older than X regardless of references): operationally similar but a different policy; defer.
- **Multi-pack-aware enumeration**: BuildLiveSet uses git rev-list which is multi-pack-aware out of the box. Nothing special needed.
- **Sweep parallelism**: serial deletes for simplicity. Bounded errgroup if benchmarks show real wall-clock pain (deferred until measured).

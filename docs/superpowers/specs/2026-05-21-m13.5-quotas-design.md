# M13.5 — Per-tenant LFS byte quotas (design spec)

**Status:** Design.
**Date:** 2026-05-21.
**Author:** Eran (with Claude).
**Predecessor:** M13.4 LFS GC (commit `c5054ed`, tag `m13.4-lfs-gc`) — unblocks the atomic-decrement-on-removal half of quota accounting.

---

## 1. Goal

Add a per-tenant hard cap on bytes stored under `tenants/<t>/repos/*/lfs/objects/`. Enforce at the LFS Batch handler before issuing upload URLs; track usage in the M4 authdb sqlite; correct drift via an operator-run reconcile command and via the M13.4 sweep path.

The OSS scope (spec §35) does not include quotas; this milestone moves bucketvcs from "OSS-feature-complete LFS" toward the commercial-product scope of spec §36, but stays inside the existing OSS process model (single gateway, sqlite authdb, no control plane).

## 2. Non-goals

- Per-repo quotas (deferred, §11).
- Git pack / index / bundle byte quotas (deferred, §11).
- Reservation system to eliminate cross-batch race (deferred, §11).
- Quota tier definitions / billing plans / chargeback (out of scope — operators wire their own billing system to call `bucketvcs quota set`).
- Soft caps / alert-only mode (use the `lfs_quota_bytes_used` gauge against the limit; operators set thresholds in their own monitoring stack).
- Webhooks on threshold crossing (spec §24 webhooks deferred independently).
- Multi-region accounting / replication of the quota counter.

## 3. Scope

**In scope:**

- New `quotas` table in the M4 authdb (sqlite migration 0004).
- New `internal/lfs/quota` package wrapping the table with Service methods.
- Batch-time pre-check at `internal/lfs/batch.go::Build`.
- Verify-time increment at `internal/lfs/handler.go::handleVerify`.
- Sweep-time decrement at `internal/lfs/gc/gc.go::RunSweep`.
- Reconcile via storage listing.
- 4 CLI subcommands: `bucketvcs quota {set,show,reconcile,clear}`.
- 2 metrics + 2 audit events.
- Localfs smoke + unit + integration + CLI tests.
- Operator-guide §8.4 flip from "deferred" to "implemented" with the trigger-condition / failure-modes / observability subsections established by §8.3 in M13.4.

**Out of scope (deferred — see §11).**

## 4. Architecture

### 4.1 State location

The quota counter is **per-tenant state on the gateway's authdb sqlite**. This is consistent with M13.3 LFS Locks (also a sqlite table on the authdb) and M4 (users/tokens/repos/permissions). Operators who already accept the M4 sqlite operational model accept the M13.5 model unchanged.

A new migration `0004_quotas.sql` adds:

```sql
CREATE TABLE quotas (
    tenant      TEXT PRIMARY KEY,
    limit_bytes INTEGER NOT NULL,
    used_bytes  INTEGER NOT NULL DEFAULT 0,
    updated_at  INTEGER NOT NULL
);
INSERT INTO schema_version (version, applied_at) VALUES (4, strftime('%s','now'));
```

No FK to `repos` — a tenant can have a quota row with zero repos (an operator pre-provisions a tier). No FK from the gateway side to enforce row presence either: **no row means no enforcement** (the M13.5-free default).

### 4.2 New package `internal/lfs/quota`

Single type `Service`. All methods take `ctx` and return an error.

```go
package quota

type Service struct {
    db     *sql.DB        // the M4 authdb handle
    logger *slog.Logger
    addRing *addRingDedupe // capacity 1024 (tenant, oid) LRU; see §6.2
}

func New(db *sql.DB, logger *slog.Logger) *Service

// CheckBatch is the Batch-time pre-check. requestedBytes is the SUM
// of all sizes in the batch. Returns ErrQuotaExceeded with details
// when (used_bytes + requestedBytes) > limit_bytes for an enforced
// tenant. Returns nil when (a) no row exists (unlimited), or (b) the
// batch fits.
func (s *Service) CheckBatch(ctx context.Context, tenant string, requestedBytes int64) error

// Add is the verify-success increment. Idempotent within the dedupe
// ring's TTL window (see §6.2). No-op when no quota row exists for
// the tenant.
func (s *Service) Add(ctx context.Context, tenant string, oid string, bytes int64) error

// Subtract is the sweep-success decrement. Floors at zero via
// MAX(used_bytes - ?, 0) to absorb reconcile-vs-sweep drift (§6.4).
// No-op when no quota row exists for the tenant.
func (s *Service) Subtract(ctx context.Context, tenant string, oid string, bytes int64) error

// Reconcile lists the LFS prefix for the tenant, sums sizes, and
// overwrites used_bytes. Returns a Report with before/after/drift.
// dryRun=true reports the drift without writing.
func (s *Service) Reconcile(ctx context.Context, store storage.ObjectStore, tenant string, dryRun bool) (Report, error)

// Set / Get / Clear are the CLI surface. Set creates or updates a
// row; Get returns the current state; Clear removes the row (back
// to unlimited).
func (s *Service) Set(ctx context.Context, tenant string, limitBytes int64) error
func (s *Service) Get(ctx context.Context, tenant string) (State, error)
func (s *Service) Clear(ctx context.Context, tenant string) error
func (s *Service) List(ctx context.Context) ([]State, error)

type State struct {
    Tenant     string
    LimitBytes int64
    UsedBytes  int64
    UpdatedAt  time.Time
    Exists     bool // false when no row (unlimited)
}

type Report struct {
    Tenant      string
    BeforeBytes int64
    AfterBytes  int64
    DriftBytes  int64 // signed: positive when actual > counter
    DryRun      bool
}

var ErrQuotaExceeded = errors.New("lfs/quota: tenant quota exceeded")

// QuotaError is the structured form returned by CheckBatch. The
// Batch handler propagates these fields into the LFS ObjectError
// 507 response so operators can debug from a single log line.
type QuotaError struct {
    Tenant         string
    CurrentBytes   int64
    LimitBytes     int64
    RequestedBytes int64
}
```

### 4.3 Integration seams

| Seam | File | Call |
|---|---|---|
| Batch pre-check | `internal/lfs/batch.go::Build()` | one `quota.CheckBatch` before the per-object loop |
| Verify increment | `internal/lfs/handler.go::handleVerify` | one `quota.Add` after the verify check returns ok |
| Sweep decrement | `internal/lfs/gc/gc.go::RunSweep` | one `quota.Subtract` per successful delete |
| CLI | `cmd/bucketvcs/quota.go` (new) | `quota.Set / Get / Clear / Reconcile / List` |

The Service is wired through `internal/lfs.Deps` (already carries Store + Logger + presigning keys). New field:

```go
type Deps struct {
    // ... existing ...
    Quota *quota.Service // nil = quota disabled (no enforcement, no counter changes)
}
```

When `Deps.Quota` is nil, every integration seam is a no-op. This makes M13.5 fully opt-in — pre-M13.5 deployments and operators who don't want quotas see zero behavioral change.

## 5. Data flow

### 5.1 Happy path

Tenant `acme` at 95 GiB used, 100 GiB limit. Client pushes 3 LFS objects (1 GiB / 2 GiB / 1 GiB).

1. `POST /info/lfs/objects/batch` (operation=upload, 3 objects, total 4 GiB).
2. `handleBatch` → `quota.CheckBatch(ctx, "acme", 4 GiB)`. SQL: `SELECT used_bytes, limit_bytes FROM quotas WHERE tenant='acme'`. (95 + 4) ≤ 100 → returns nil. Metric `lfs_quota_check_total{outcome=ok}` fires.
3. `Build()` issues upload + verify URLs for all 3.
4. Client PUTs each object.
5. Client POSTs verify per object. `handleVerify` → `quota.Add(ctx, "acme", oid, size)`. SQL: `UPDATE quotas SET used_bytes = used_bytes + ?, updated_at = ? WHERE tenant = ?`. Counter: 95 → 96 → 98 → 99 GiB.
6. (Later) M13.4 LFS GC runs. Sweep deletes an unreferenced 5 GiB object. `RunSweep` → `quota.Subtract(ctx, "acme", oid, 5 GiB)`. Counter: 99 → 94 GiB.

### 5.2 Quota-exceeded path

Same tenant tries to push one 10 GiB object after step 5.

1. Batch operation=upload, 1 object, size=10 GiB.
2. `quota.CheckBatch("acme", 10 GiB)`. (99 + 10) > 100 → returns `QuotaError{Tenant: "acme", CurrentBytes: 99 GiB, LimitBytes: 100 GiB, RequestedBytes: 10 GiB}`.
3. Every Upload object in the Batch response gets:
   ```json
   {"oid": "...", "size": ..., "error": {"code": 507, "message": "tenant quota exceeded: 99GiB used / 100GiB limit, 10GiB requested"}}
   ```
4. Metric `lfs_quota_check_total{outcome=exceeded}` + audit `lfs.quota.exceeded` fire.
5. Client receives 200 OK Batch with per-object 507 errors; `git lfs push` surfaces the error message to the user.

### 5.3 Reconcile

Operator runs `bucketvcs quota reconcile --tenant=acme`.

1. Service.Reconcile reads current `used_bytes` (= `before`).
2. Lists `tenants/acme/repos/*/lfs/objects/` (uses Storage.ObjectStore.List with pagination, sums sizes across all repos).
3. Computes `after` = sum.
4. If `dryRun=false`: `UPDATE quotas SET used_bytes = ?, updated_at = ? WHERE tenant = 'acme'`. Else: no write.
5. Returns `Report{Tenant, BeforeBytes: before, AfterBytes: after, DriftBytes: after - before, DryRun: dryRun}`.
6. Metric `lfs_quota_bytes_used{tenant=acme}` = `after` is emitted. Audit `lfs.quota.reconcile` fires.

## 6. Race semantics and drift

### 6.1 Within-batch race

CheckBatch sums all object sizes in a batch atomically against the current counter. One SQL SELECT per batch — no race within a single batch request.

### 6.2 Verify replay (idempotency of Add)

M13.1's kind=5 verify token is bound to `(tenant, repo, oid)` with a TTL; it is NOT consume-on-use. A client may legitimately re-POST verify within the TTL (network retry, transient timeout). `quota.Add` keeps an in-memory ring (capacity 1024 entries, LRU eviction) of `(tenant, oid)` pairs seen recently. Re-adds within the ring are no-ops.

**Beyond ring eviction**, double-counting may occur. Bound: `~1024 distinct (tenant, oid) pairs × max_per-object overhead`. For a typical gateway pushing thousands of distinct objects/hour, eviction may occur within a verify TTL window. The reconcile is the operator's safety net.

A persistent dedupe (sqlite row per `(tenant, oid)`) was considered and rejected: it would double the verify-path's DB cost for a race that reconcile already handles cleanly.

### 6.3 Cross-batch race

Two concurrent batches for the same tenant at 99/100 GiB, each pushing 2 GiB:

- Batch A: CheckBatch reads used=99, accepts (99+2 ≤ 100).
- Batch B: CheckBatch reads used=99 (A hasn't verified yet), accepts.
- Both upload, both verify-increment. Counter ends at 103 GiB.
- Next batch from this tenant: CheckBatch reads used=103, rejects anything > 0 — quota is now correctly hard-capped from this point forward.

Bound: `concurrent_batches × max_batch_size`. Documented as best-effort. Reconcile fixes if the operator cares; the next push from the over-tenant is correctly rejected regardless.

A reservation system that holds bytes from CheckBatch until the verify (or TTL expiry) was considered. Rejected: requires a TTL state machine, complicates failure recovery, and operators get the same effective behavior by setting the limit slightly under the contractual ceiling.

### 6.4 GC + reconcile interaction

A sweep happening mid-reconcile could decrement after reconcile has read `used_bytes` but before reconcile writes back. The reconcile uses storage as authoritative — it overwrites with the listing sum. Any decrement that landed during the scan window is captured by the listing (the object is gone, the listing doesn't see it, the sum is correct).

The `MAX(used_bytes - ?, 0)` clamp on Subtract handles the reverse race: reconcile already corrected drift down, then a sweep decrements further — counter would go negative. The clamp keeps it at 0; the next reconcile re-establishes truth.

### 6.5 Backend errors during Add / Subtract

An authdb write failure during verify-success is logged at ERROR but does NOT fail the verify response to the client. The client already uploaded; failing verify would cause the client to retry the upload, wasting bandwidth and possibly violating idempotency assumptions in the LFS spec.

Same for Subtract during sweep: the delete already happened; failing the sweep would re-attempt the delete on the next mark, which is fine but wasteful.

Drift from these failures is captured by the next reconcile. The trade is **correctness-of-counter < availability-of-LFS** — same trade-off the operator guide §1 documents for ephemeral state under M11/M13.

## 7. CLI

New subcommand tree mounted in `cmd/bucketvcs/main.go::dispatch`:

```
bucketvcs quota set       --auth-db=<path> --tenant=<t> --limit=<size>
bucketvcs quota show      --auth-db=<path> {--tenant=<t> | --all}
bucketvcs quota reconcile --auth-db=<path> --store=<URL> {--tenant=<t> | --all-tenants} [--dry-run] [--format=text|json]
bucketvcs quota clear     --auth-db=<path> --tenant=<t>
```

### 7.1 Common flags / behavior

- `--auth-db=<path>` is required on every subcommand (consistent with the existing M4 `bucketvcs user/token/repo` commands).
- `--store=<URL>` is required on `quota reconcile` only — reconcile lists the LFS storage prefix; set/show/clear touch only the authdb.
- Size suffixes accepted on `--limit`: `B`, `KB`/`KiB`, `MB`/`MiB`, `GB`/`GiB`, `TB`/`TiB`. Binary (1024) and decimal (1000) supported with the standard distinction; bare numbers are bytes. Rejects negative values.
- Exit codes: 0 ok, 1 operational error, 2 usage error.
- `--format=text` (default) is human-readable; `--format=json` emits one JSON record per tenant for `show --all` / `reconcile --all-tenants`.

### 7.2 `quota show` output

When `used_bytes > limit_bytes` (e.g., the operator just lowered the limit, or reconcile revealed drift), the text output includes `over_by=<bytes>`:

```
tenant=acme  used=110GiB  limit=100GiB  over_by=10GiB  updated=2026-05-21T12:34:56Z
```

JSON form carries the same `over_by_bytes` field (zero or positive; never negative).

### 7.3 `quota reconcile` output

Text:
```
tenant=acme  before=99GiB  after=98.7GiB  drift=-0.3GiB  (counter was over-reporting)
```

JSON:
```json
{"tenant":"acme","before_bytes":...,"after_bytes":...,"drift_bytes":...,"dry_run":false}
```

`drift_bytes` is signed: positive when storage exceeded the counter (likely cause: an Add was dropped on a transient authdb error); negative when the counter exceeded storage (likely cause: a Subtract was dropped, or an OOB deletion bypassed GC).

## 8. Observability

### 8.1 Metrics (additions to `internal/lfs/metrics.go`)

- `lfs_quota_check_total{outcome}` — counter.
  - `outcome=ok`: batch fit the quota (or no quota row exists for this tenant).
  - `outcome=exceeded`: batch was rejected.
  - One emission per Batch pre-check.

- `lfs_quota_bytes_used{tenant}` — gauge (post-update value, not a delta).
  - Refreshed on every successful Add, Subtract, Reconcile, Set, Clear.
  - Operators graph this directly; alert when `value / limit > 0.9` (soft-cap proxy).

Both exported (capital-E prefix) because the integration seams live in sibling packages (`internal/lfs` for Batch/verify, `internal/lfs/gc` for sweep) — same convention as M13.4's `EmitGCObjectsMarkedMetric` family.

### 8.2 Audit events (additions to `internal/lfs/audit.go`)

- `lfs.quota.exceeded` — fires on every rejected Batch.
  ```
  event=lfs.quota.exceeded
  tenant=<t>
  current_bytes=<int64>
  limit_bytes=<int64>
  requested_bytes=<int64>
  oids=[<oid>, ...]   // the rejected batch's OIDs, comma-separated
  ```

- `lfs.quota.reconcile` — fires once per reconcile call.
  ```
  event=lfs.quota.reconcile
  tenant=<t>
  before_bytes=<int64>
  after_bytes=<int64>
  drift_bytes=<int64-signed>
  dry_run=<bool>
  ```

No audit on Add / Subtract — those happen in the hot path and are already covered by the existing `lfs.object.served` and `lfs.gc.sweep` events.

## 9. Failure modes

| Symptom | Cause | Action |
|---|---|---|
| Batch returns 507 on every object | Tenant at or above quota | `bucketvcs quota show --tenant=<t>` and either raise the limit or wait for GC |
| Tenant exceeds quota despite enforcement | Cross-batch race (§6.3) | `bucketvcs quota reconcile --tenant=<t>` to update the counter; consider lowering the limit by `concurrent_batches × max_object_size` as a buffer |
| Reconcile shows persistent drift | Add/Subtract write failures going unlogged | Check authdb error logs; reconcile is the safety net |
| Reconcile shows large positive drift after a fresh deployment | Counter started at zero, existing LFS objects pre-exist | Expected — run reconcile once after first M13.5 deploy to seed the counter |
| `quota show` reports `over_by=<bytes>` | Limit was lowered below current usage, or drift caught up | New uploads correctly rejected until GC + reconcile drain usage |

## 10. Migration / compatibility

### 10.1 Schema

- `0004_quotas.sql` adds the table. Forward-only.
- The M4 `RunMigrations` infrastructure already applies sequenced migrations; no changes there.
- Removing M13.5 means dropping the quota CLI and not wiring `Deps.Quota`; the table is harmless when unused. A future migration could drop it.

### 10.2 Backward compatibility

- A pre-M13.5 gateway running against an M13.5-migrated authdb works unchanged — it doesn't query the quotas table.
- An M13.5 gateway running against a pre-M13.5 authdb runs `RunMigrations` on startup (existing behavior), bumping schema 3→4 in-place.
- An M13.5 gateway with no `Deps.Quota` (operator opts out) behaves exactly like pre-M13.5 — no enforcement, no counter changes, no metric emissions for the quota family.

### 10.3 Bootstrap

After a fresh M13.5 deployment with pre-existing LFS objects in storage:

```bash
# One-time bootstrap: seed counters for every tenant.
bucketvcs quota reconcile --all-tenants
```

This is documented in operator-guide §8.4 as the first step after upgrade. The reconcile is idempotent — re-running it overwrites with the current listing sum.

## 11. Deferred (explicit non-goals)

- **Per-repo quotas.** Second table `repo_quotas (tenant, repo, limit_bytes, used_bytes, updated_at)` with the same Service shape. Defer until operator pain materializes; the per-tenant cap is the load-bearing case for billing/chargeback.
- **Git pack / index / bundle byte quotas.** Requires hooking receive-pack + maintenance.Materialize + bundle generation. Roughly 3× the integration surface of LFS quotas for marginal value (Git bytes are <10% of repo bytes when LFS is in use). M13.6+ if ever needed.
- **Reservation system to eliminate the cross-batch race.** Documented in §6.3 as best-effort; reconcile handles drift; operators get the same effective behavior by setting the limit slightly under the contractual ceiling.
- **Quota tiers / plan definitions / chargeback.** Lives in a control plane outside the gateway. Operators wire their billing system to call `bucketvcs quota set` on plan changes.
- **Soft-cap / alert-only mode.** Operators set thresholds on the `lfs_quota_bytes_used` gauge in their own monitoring stack; the gateway exposes the data, not the alert policy.
- **Grace period.** Same reasoning — set the limit higher if headroom is desired.
- **Webhook on threshold cross.** Webhooks themselves are deferred (spec §24).
- **Multi-region / replicated quota counter.** Multi-region (spec §26) is deferred independently.
- **Real-time decrement on direct-storage deletions** (lifecycle rules, manual S3 deletes). Reconcile is the only correction mechanism for OOB deletions.
